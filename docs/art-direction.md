# Art Direction

## Read

Independent top-down 2D pixel arena, visually aligned with the supplied Vinci World screenshot: cozy village RPG, crisp 32-bit pixel texture, bronze HUD frames, warm houses, stone paths, lush grass, and a central rainbow-glass Renaiss/Vinci logo fountain.

## Production Rules

- Keep the game screen as the primary surface. No marketing landing page before the game.
- Pixel art must use consistent outline weight, lighting angle, and top-down RPG proportions.
- UI chrome uses compact game HUD panels, not generic web cards.
- Avoid AI-looking gradient trim lines, click sparks, boxed card stacks, persistent stand rings, and card-back platforms behind combat pets.
- Combat readability wins over decoration: projectiles, hit flashes, health bars, and cooldowns must be legible.
- Web implementation target: Phaser/WebGL for game rendering, React for code-native HUD and menus.
- RPG dojo battle backgrounds must be static premium retro pixel art. Do not add reflective/glass floors, noisy animated background movement, non-pixel wall decorations, or side-view scenery pasted behind the pets.

## Generated Assets

Concept and asset sheets were generated through the built-in image generation tool and copied into `apps/client/public/assets/generated`. Runtime code removes the flat chroma backgrounds from the sheets before drawing sprites and environment props.

Class sprites are normalized from `tools/assets/generated-sources/class-sprites-source.png` by `tools/normalize_class_sprites.py`. Keep all four classes in the same high-detail top-down RPG sprite style; do not replace a broken class with a lower-detail programmatic redraw unless the whole class set is redrawn together.
Mage front idle frames 0-2 use `tools/assets/generated-sources/mage-front-repair-source.png` because the original source hood touched the top edge and read as clipped in the class portrait. Repair the full hood/head silhouette from a regenerated source; do not patch this with loose bangs, hair pieces, or frame-edge pixel stickers.
Mage side-walk frames 4-6 come from the generated three-frame source `tools/assets/generated-sources/source/mage-side-walk-generated-source.png`, produced from the approved front Mage reference. Do not rebuild them by patching the old broken side-walk reference or by removing swords from the legacy source walk bodies. Keep frames 4-6 as walk/idle frames, with frame 7 reserved for the attack cast pose. Frame 7 must keep a transparent safety margin so the cast pose does not read as clipped in runtime slicing.
If any class portrait looks blocky or simpler than the other three, treat it as an asset-source or browser-cache problem first. Do not solve it with a hand-drawn low-detail substitute; regenerate from the high-detail sheet and bump the generated asset version.
When a source sprite has enclosed chroma-key holes, such as Archer's bow frame, remove the internal high-saturation green matte as well as border-connected green. Do not solve enclosed matte leaks by cropping the full sprite tighter.

Skill icons are deterministic 64px pixel cells generated by `tools/generate_skill_icons.py`. Each icon must be extracted from one complete production source row or character action row, then normalized into the 4x4 class/slot grid. Do not return to programmatic hand-drawn composites, mixed-source overlays, or large painterly source art scaled down as a shortcut.

Combat VFX sheets are first generated by `tools/generate_pixel_vfx.py`, then arena rows are upgraded by the Combat FX, SpellsFX, and Super Pixel import scripts. RPG skill and projectile sheets are upgraded through `tools/import_external_rpg_vfx.py`. The RPG importer chooses one complete external frame sequence per skill row from `SpellsFX2.0.zip` or `Super Pixel Effects Gigapack (Free Version) v2.5.0.zip`, and normalizes `New_All_Fire_Bullet_Pixel_16x16.rar` into the 5-row RPG projectile sheet, so runtime code keeps clean spritesheet contracts.
External impact VFX are used for the main move hit/readability layer: slashes, bursts, spell silhouettes, rain, fields, beams, and summon-style finishes. A single skill row must not combine A+B+C effects from multiple packs or multiple sequences. The paid external 16x16 fire-bullet atlas is used only for small actor-to-target travel projectiles; do not stretch it into ultimate or field effects. Gigapack effects should stay small and crisp; do not upscale explosion frames into blurred full-screen blobs.
RPG skill VFX production mapping lives in `packages/shared/src/rpgVfxProduction.ts`. Keep this mapping authoritative: small fast shots such as 星火彈/種子飛鏢/夜針飛射/晨星彈 use the paid external 16x16 bullet atlas as their one projectile source and must not add a second caster, target, or ground-impact skill sheet on top. Other skill rows use exactly one complete SpellsFX or Gigapack sequence as the hit/support/group/ultimate read. If a move travels from actor to target, that travel sprite is the only primary skill layer; do not also render a target impact copy, windup copy, or finish copy from the same or another pack. Persistent status overlays may appear as separate `rpg-status-vfx.png` rows, but each status row is also one complete sequence, not a blend. Non-ultimate self, single-ally, and all-allies moves must stay `support-field`, never enemy-style status or sweep categories.
The runtime contract separates `primarySource` from `statusSource`: `data-vfx-source` must name exactly one main skill sequence, and `data-vfx-status-source` is only for persistent burn/poison/guard/regen-style overlays. Do not use a comma-separated source list to justify mixing two main skill packs in one move.
Battle hits must land on the field through the selected spritesheet itself, not through extra CSS-drawn stickers, rings, slashes, or ground-contact marks. Do not reintroduce `rpg-fx-ground-impact` or `rpg-skill-ground-impact`; group attacks should read by placing the chosen external sequence on each affected target.
Do not reintroduce the old hand-authored CSS skill particles (`rpg-fx-core`, `rpg-fx-sweep`, `rpg-fx-pixel`) or status pseudo-layers. Visible skill and status art should come from generated spritesheet/projectile components; CSS may position, mirror, time, or fade those sprites but should not draw a replacement effect.
In versus battles, VFX placement must follow the viewer's local presentation: the current player's team is rendered on the left even when their server-side pet objects originally belong to the right seat. Do not use the raw pet `side` field for visual coordinates after a battle snapshot has been mirrored for the right player.
Projectile and skill-icon assets must read as finished RPG pixel assets at runtime size: thick dark silhouettes, layered highlights, and a clear body/trail relationship. Avoid thin-line projectiles that only become visible by being oversized in Phaser.
Persistent status effects use `rpg-status-vfx.png` as the runtime source. The status review page must show burn, poison, stun, guard, and regen as separate rows, with every status retaining a distinct frame strip and field overlay.
Skill previews for status moves must also show those persistent status layers on affected pets. A burn, poison, stun, guard, or regen move is not visually complete if the review page only shows impact VFX and text.
Support previews need their own read: healing/cleanse previews use the regen row as the visible recovery layer, guard/shield previews use the guard row, and affected allies must not play hit poses or enemy-impact shakes.
Before release, keep proof of commercial-use rights for any external VFX pack in the project license record. The BDragon RPG VFX packs are marked `commercial-proof-confirmed` because the project owner confirmed payment and commercial-use approval in the Codex thread on 2026-07-03. SpellsFX 2.0 is release-cleared through its included CC BY 4.0 license and required Raphael Hatencia / RagnaPixel Studio attribution. Super Pixel Effects Gigapack Free Version is release-cleared through its included license and required Will Tice / unTied Games attribution. The external VFX importer preserves a complete confirmed proof block during regeneration; incomplete proof is intentionally reset to pending.
Basic melee attacks must not share one oversized white slash. Warrior attacks use a compact sword arc, Engineer attacks use a smaller mechanical spark strike, and both should leave the character silhouette readable.

The RPG battle field is generated by `tools/generate_rpg_pet_sprites.py` as `rpg-battle-arena.png`. It should read as a built battle venue with grass-framed stone courtyard texture, a floor-printed Vinci favicon mark at center, village-aligned pixel detail, curb/brace dressing, edge props, and subdued battle wear. It must not read as a pure slab or pasted side-view dojo. Keep the field as an opaque 1600x900 PNG with crisp 2x pixel blocks; `pnpm assets:validate` checks this asset and rejects a return to baked slot rings, court crosslines, or cropped edge props.
Battle-field pets stand directly on the field. Do not add card stacks, flat platforms, persistent stand rings, decorative mats, baked floor scuffs, or CSS drop-shadow/glow filters behind them; the pet silhouette and spritesheet VFX should carry the read. `pnpm assets:validate` also rejects detached thin ground decorations inside the pet spritesheet so stand rings and floor arcs cannot return through generated frames, and the release/browser gates reject battle-field pet pseudo-elements or runtime drop-shadows.

Starter pet silhouettes must remain readable when scaled down in the 3v3 arena: water needs fins and wave trails, fire needs a sharp flame back, grass needs branch antlers and leaf clusters, dark needs crescent/smoke shapes, and light needs wing feathers with halo pixels. `pnpm assets:validate` checks these element signature regions on the generated pet sheet.

Combat objects live on a top-down arena floor. Turrets should use a grounded shadow/anchor with a top-down rotating body; do not ship side-view cannon art that looks pasted onto the stone path when the barrel aims.

The class portrait canvas keeps a higher internal resolution than its visible CSS size. If a portrait looks like a low-detail blocky redraw while the sheet is high-detail, inspect the canvas backing size and cache version before changing the source art.

Field recovery pickups should read as compact Renaiss recovery props with the current logo, a grounded shadow, and a restrained glow variant from `healthPackVariants.ts`. Do not bring back the old field-sigil concept or oversized map crystals.
